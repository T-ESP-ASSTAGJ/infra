# Ansible Lab Notes

The Ansible subtree is preserved here as a standalone lab environment rather than an integrated deployment path.

## Main Entry Points

- `playbooks/gateway_server_setup.yml`: OpenBSD gateway bootstrap
- `playbooks/pki_server_setup.yml`: dedicated PKI Vault bootstrap
- `playbooks/vault_server_setup.yml`: separate secrets Vault bootstrap
- `playbooks/master_node_setup.yml`: RKE2 control-plane and cluster foundations
- `playbooks/database_node_setup.yml`: service integration, issuers, databases, and secret wiring

## Vault and PKI Model

This lab intentionally separates certificate authority duties from secret-management duties:

- `pkiserver` runs Vault as the PKI authority
- `vaultserver` runs a second Vault instance for application secrets and dynamic credentials

That split is referenced directly by `database_node_setup.yml`, which talks to both Vault endpoints when configuring Kubernetes auth, issuers, and secret engines.

## Expected Local Inputs

- `vaulted_secrets.example.yml` documents the variables that were previously provided via an encrypted file
- `playbooks/rke2_join_info.example.yml` documents the worker join variables used by the RKE2 worker setup task
- `network_vars.yml` contains sample values generated by `scripts/network_setup.sh` and should be adapted per host

## Important Constraints

- inventory targets a Vagrant-based lab layout
- several manifests still contain placeholder endpoints or credentials and are not production-ready
- Vault initialization writes local key material during execution; those artifacts are ignored by `rnd/.gitignore`
- some flow assumptions are encoded directly in playbook ordering rather than wrapped in a single orchestrator
